AuditLayer Media · operating architecture

One canonical ALM operator, four bounded runtime roles.

A plan for a small founder-led company where Narin and Ashesh share one ALM agent front door, while report generation, coding, operations, monitoring, and production changes remain isolated, auditable, and safe.

Prepared 21 July 2026 Repository: ~/projects/auditlayer Planning only — no profile, bot, schema, or service changes made

Recommended decision. Create a new Hermes profile named alm as the single human-facing company operator. Give it the new Telegram bot and expose it inside the authenticated ALM admin report workspace. Keep three hidden execution profiles: alm-dev, alm-ops, and alm-report.

The shared source of truth is not a mutable profile directory. It is the versioned hermes-profile/ package in the AuditLayer repository. Profiles are installed from that package. Product report runs use isolated per-account Hermes homes seeded from the same package and remain hard-pinned to DeepSeek V4 Flash.

1. Why one literal shared profile is unsafe

Hermes profiles isolate configuration, API keys, memory, sessions, skills, cron jobs, and gateway state. They do not sandbox the filesystem. A Telegram conversation, two concurrent report jobs, a Sentry incident, and a production deployment should therefore not write into the same mutable Hermes home.

The right unification boundary is one canonical ALM identity and one versioned instruction bundle, not one shared state database.

Do not point the report worker at ~/.hermes/profiles/alm. Concurrent jobs could mix sessions and memories, interactive changes could alter production behavior, and the broad operator toolset would leak into automated report generation. The existing account-scoped HERMES_HOME mechanism is a strong foundation and should be retained.

Why not separate frontend and backend profiles? In a two-founder company, that creates handoff overhead and duplicated context. Use one alm-dev owner for coherent full-stack outcomes. For genuinely parallel work, the operator may launch two temporary worktrees or bounded subagents labeled frontend/backend; those are execution lanes, not permanent company identities.

Why Sentry is not a profile. Sentry is an event source and evidence system. alm-ops consumes its signals. A permanent “Sentry agent” would duplicate the operations role and create another memory/config surface.

7. Admin report workspace

The report page should gain an admin-only “ALM Operator” panel. This is not exposed to clients and is not a replacement for the existing section-scoped client refinement form.

Figure 2. Discussion and evidence remain adjacent to the report. Mutating actions are explicit rather than inferred from free-form chat.

Frontend flow

  1. Server component re-checks profiles.role = 'admin'.
  2. Create or resume a deterministic Hermes session such as alm:report:<audit_id>.
  3. Inject a bounded report context envelope: audit metadata, current version, limitations, recent events, requested section, and signed-on-demand artifact access. Never place a long-lived signed URL in the database.
  4. Stream the operator response into the panel and persist a product-side audit trail.
  5. Discussion is read-only by default. “Queue refinement,” “Create engineering task,” “Apply fix,” and “Deploy” are separate typed actions with permissions and confirmations.

Secure connection pattern

Do not expose the Hermes API key to the browser. The Next.js server route should call the profile-scoped endpoint /p/alm/v1/chat/completions through a dedicated Cloudflare Tunnel origin protected by a Cloudflare Access service token and Hermes bearer key. The VM listener remains firewall-blocked from the public internet except through the tunnel.

For long-running jobs, return an operator job ID and continue on the VM rather than keeping a Vercel stream open indefinitely. Supabase Realtime can carry progress and completion events back to the admin workspace.

8. Sentry and operational signals

Add two Sentry projects or one project with strict environment tags:

  • Web: Next.js server, edge, route handlers, and client errors; release tied to Git SHA and Vercel deployment.
  • Worker: Python exceptions, failed audit/refinement IDs, queue phase, worker instance, prompt/template version, and release SHA. Never attach creator payloads or secrets by default.

Normalize Sentry issue alerts through a signed webhook endpoint. The endpoint should deduplicate on Sentry fingerprint, scrub data, record an operator_incident, and trigger alm-ops with a fixed prompt template. Do not pass the raw webhook body to an agent as instructions.

9. The iterative fixing loop

Figure 3. The operator owns state and approvals; specialist profiles own bounded execution. Closure requires production evidence, not a model’s assertion.

  1. Observe: Sentry, health probes, report feedback, or founder message creates a normalized incident/task.
  2. Triage: alm-ops reads current evidence, identifies blast radius, and chooses report/content, frontend, backend, data, or infrastructure lane.
  3. Reproduce: alm-dev or alm-ops creates a deterministic failing test or verified runtime probe before changing code.
  4. Fix: one owner implements the coherent outcome in a branch/worktree. Parallel agents are used only for genuinely disjoint work.
  5. Verify: run targeted tests, make check-v2, visual/browser QA when relevant, security checks, and model-free report fixture checks.
  6. Approve: Narin approves report/product quality changes; Ashesh approves code and production changes. High-risk changes require both when they affect the report contract.
  7. Release: deploy, wait for settling, verify public endpoint and both workers, inspect Sentry/restart counters, and record the result in durable release/incident tables.
  8. Learn: update a repo skill/runbook only for reusable procedure. Do not store SHAs or completed incident logs in profile memory.

12. Release gates

  • Identity: only Ashesh and Narin can reach the Telegram bot; browser operator routes are admin-only.
  • Isolation: operator, dev, ops, and report runtime have separate state; account report homes remain tenant-isolated.
  • Model: report runtime proves deepseek-v4-flash/deepseek with no Codex fallback.
  • Tools: report runtime cannot call terminal, messaging, file writes, delegation, deployment, or company memory.
  • Observability: web and both worker instances emit environment/release-tagged Sentry events with PII scrubbing.
  • Auditability: report changes, operator jobs, incidents, approvals, and releases have durable database/Git evidence.
  • Frontend security: no Hermes, Supabase service-role, Cloudflare service, or Sentry secret reaches client JavaScript.
  • Quality: make check-v2, targeted worker tests, admin Playwright coverage, visual QA, and post-deploy probes pass.
  • Recovery: rollback is documented and exercised before legacy profile retirement.

15. Definition of done

The setup is complete when Narin can message one Telegram bot or open any report as an admin, discuss the report with ALM’s durable company context, and queue a safe explicit refinement; Ashesh can route a frontend, backend, or operations problem through one operator; Sentry creates deduplicated evidence-backed incidents; report generation consumes the same versioned ALM knowledge bundle while remaining tenant-isolated and DeepSeek V4 Flash-only; and every code/report/production change passes an approval and verification trail.
